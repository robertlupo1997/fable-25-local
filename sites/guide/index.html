Project Guide · 2026

How FABLE 25
Local Was Built

Twenty-five websites. One model. Everything that worked, everything that failed, and the pattern that got us across the line.

The premise

In 2025, Nick Saraev published a video in which he watched Claude Fable 5 autonomously build 25 websites in an afternoon — one prompt per site, one agent per site, parallel. The result was fable-25.netlify.app. This project is inspired by that work, not part of it and not affiliated with it. The original prompt (transcribed from his video) is preserved verbatim in PROMPTS.md.

The question we tried to answer differently: what happens when you apply the same method with original briefs, on a single local machine, with no deployment pipeline, no image-generation APIs, and a requirement that every site's visual effects be earned through code — GLSL shaders, canvas simulation, SVG, CSS, WebAudio synthesis, and math?

This is not Nick's project. The 25 concepts here are original. The method is adapted. Every prompt we issued is in PROMPTS.md; every brief is in BRIEFS.md. Nothing in this repo should be taken as representative of his project or his results.

What's different from the original

Local, not deployed-first

Everything builds and serves on one machine. This repo is the publication. No Netlify, no Vercel, no CI. A Python HTTP server at port 4179. This meant QA had to happen locally too — a Playwright-based screenshot harness (tools/shot.mjs) that renders real pages in headless Chromium, returns the screenshot as a file, and surfaces any JavaScript console errors or page errors in its JSON output.

Every pixel is procedural

No image-generation APIs were available. So the 25 briefs were art-directed specifically around what code can do: Rosensweig instability simulations, thin-film interference shaders, moth flight physics, earthquake data essays, atmospheric scroll journeys, shadow-puppet SVG theatre. No stock images, no AI-generated illustrations, no templates, no shared components.

Computational contrast verification

WCAG AA was verified computationally, including pixel-sampling rendered scenes behind text — not just reading CSS color values. When a site layers text over a live canvas or WebGL scene, the background color at the text position is sampled from the actual rendered pixels. Each site's /guide documents what was verified and what wasn't.

Original briefs

The 25 concepts are new: an aurora research station, a numbers-station archive, a ferrofluid laboratory, a century of real USGS earthquakes, a museum of impossible objects, a polyrhythm observatory. The full set is in BRIEFS.md.

The method

Step one — design constitution + 25 briefs

An orchestrator session wrote a design constitution (the non-negotiable build standard, reproduced in full below) and 25 individually art-directed briefs. Each brief specifies: the concept and its emotional register, a color palette with OKLCH values, one or two Google Fonts, a signature technique that must appear in the code, and one thing to prove — a capability claim the site must demonstrate convincingly.

Step two — parallel builder agents

Parallel Claude Fable 5 subagents — one per site — each received a single brief and full autonomy over a single folder (sites/<slug>/). No shared dependencies, no build steps, no inter-site communication. Each builder wrote its own HTML, CSS, and JS from scratch.

Step three — the three-pass screenshot critique loop

Every site's build cycle ran a minimum of three critique passes. A pass is not "look at the code." A pass is:

# Render — from /home/tlupo/projects/fable25: node tools/shot.mjs http://localhost:4179/<slug>/ \ qa/<slug>/pass1-desktop-top.png 1440x900 3500 # Then shoot mid, bottom, and mobile variants. # Check the JSON output for errors and pageErrors. # Read at most three screenshots with vision. # Critique like a hostile design director. # Fix everything found — then add one deliberate complexity upgrade.

The complexity-upgrade rule was the design decision that mattered most. Without it, critique passes converge on removing things — tightening, simplifying — until the site is bland but defensible. The rule forces the builder to add: a texture, a micro-interaction, marginalia, an easter egg, a second animation layer. It stops iteration at safe.

Step four — cold director review between waves

After each wave, the orchestrator reviewed every hero shot cold — without having seen the build process — and sent sites back with specific findings. "The number station's red is close to the error state on the radar. Change it." "The moth field guide has dead white space at 390px — the lamp should recompose." Sites went back into a polish pass before wave closeout.

The failure history

The failures were the reproducible part. Here is what actually went wrong, in the order it happened.

Failure 1 — Wave 1, four builders dead

Context overflow from undisciplined screenshot reading. Early builders shot screenshots at 2× device pixel ratio, then read every angle with vision — 6 to 8 screenshots per pass, three passes. By pass two, the context window was full of pixel data. The agent's working memory ran out mid-task, leaving sites half-finished in states that couldn't be resumed cleanly.

Fix: Cap screenshots at 1× DPR. Vision-Read at most three screenshots per pass — pick the most informative angles. The rule went into the build standard and held for all subsequent waves.

Failure 2 — Text accumulation in long-lived agents

Single agents running a full site lifecycle (build + polish + guide + closeout) accumulated too much conversation text, even without the screenshot problem. Prose replies, code blocks, tool call results — by the closeout stage, the context was degraded enough that agents made contradictory changes, forgot constraints, or produced generic copy that contradicted what earlier in the session they had written specifically.

Fix: The relay pattern. Each site's lifecycle is split across three sequential scoped agents, each handed a NOTES.md baton containing only what the next stage needs. Build agent → Polish + Guide agent → Gated Closeout agent. Zero agent deaths after the switch.

Failure 3 — One spend-limit outage mid-project

One API spend-limit interruption mid-wave. One site was mid-build. The build agent's partial state was recoverable from the working files; the NOTES.md baton held enough context that a fresh agent could continue. The relay pattern absorbed the outage cleanly — which is to say, it was designed for exactly this.

The build standard

Every site had to clear every item before closeout. Non-negotiable. These are paraphrased from the full constitution in BRIEFS.md.

  • 1Static and self-contained. One folder: sites/<slug>/ with index.html, optional style.css and main.js, small assets. CDN allowed only for Three.js and Google Fonts.
  • 2Real copy only. Fictional brands welcome, written like the best copywriter you can be — specific, confident, zero lorem. Real-world data cited in-page.
  • 3Type. One or two Google Fonts per brief, set to a deliberate scale. Kill widows in display text.
  • 4Custom easing. Cubic-bezier or spring math; never the CSS ease/linear defaults on a hero move. Every animation honors prefers-reduced-motion: reduce with a dignified static state.
  • 5Responsive 390 → 1600. Recompose for mobile — don't shrink the desktop layout.
  • 6Accessibility. Semantic landmarks, exactly one <h1>, alt text, keyboard-reachable interactions, visible focus, WCAG AA text contrast (4.5:1 normal / 3:1 large). Decorative canvases aria-hidden="true".
  • 7Zero console errors. The harness reports them; they are release blockers.
  • 8Performance sanity. No single asset over 300 KB. rAF loops pause on document.hidden. Canvas DPR capped at 2.
  • 9The /guide route. sites/<slug>/guide/index.html, written in the site's own visual language: the brief it received, its signature technique with real code, and what each critique pass found and changed. Honest, specific.
  • 10Credit line. Every footer: Designed & built by Claude Fable 5 · local build · 2026.

The collection

Twenty-five sites, four waves. Wave 1 was seven sites — the proving ground where the screenshot-reading rule and the relay pattern were discovered. Waves 2 through 4 ran with those lessons in place. Every site documents its own build process at /<slug>/guide/.

The brands are fictional and written as fiction. The real datasets are real and cited in-page: the earthquake essay uses USGS Earthquake Hazards Program FDSN catalog data; the undersea cable atlas uses published cable network data.
