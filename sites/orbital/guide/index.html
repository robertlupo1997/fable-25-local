Signature Technique: Keplerian Propagation

Every satellite position is computed from six orbital elements — semi-major axis, eccentricity, inclination, right ascension of ascending node (RAAN), argument of perigee, and mean anomaly — using the real two-body orbital mechanics model. At each animation frame, mean anomaly is advanced by the mean motion n = √(GM/a³), then Kepler's equation is solved by Newton-Raphson iteration to get the eccentric anomaly, then the satellite's 3D position is computed via the perifocal frame rotation.

The Earth also rotates beneath the satellites at the correct sidereal rate (ω_E = 7.2921 × 10⁻⁵ rad/s), so the ground track projection is correctly computed in the Earth body frame rather than ECI space.

/* Newton-Raphson solution of Kepler's equation: M = E − e·sin(E)
   6 iterations gives sub-arcsecond accuracy for near-circular LEO orbits */
function solveKepler(M, e, iters = 6) {
  let E = M;
  for (let k = 0; k < iters; k++) {
    E = E - (E - e * Math.sin(E) - M) / (1.0 - e * Math.cos(E));
  }
  return E;
}

function keplerPos(sat, t_s, out) {
  const n = Math.sqrt(GM / (sat.a * sat.a * sat.a)); // mean motion rad/s
  const M = ((sat.m0 + n * t_s) % (2 * Math.PI) + 2 * Math.PI) % (2 * Math.PI);
  const E  = solveKepler(M, sat.e);

  // True anomaly from eccentric anomaly
  const nu = 2 * Math.atan2(
    Math.sqrt(1 + sat.e) * Math.sin(E * 0.5),
    Math.sqrt(1 - sat.e) * Math.cos(E * 0.5),
  );

  // Perifocal frame → ECI cartesian (X toward vernal equinox, Z = north pole)
  // R₃(−Ω)·R₁(−i)·R₃(−ω) rotation applied
  const r  = sat.a * (1 - sat.e * Math.cos(E)) * SCALE;
  // ... P/Q direction cosines computed from (ω, i, Ω) ...
  // Three.js Y-up = ECI Z (north pole); Three.js −Z = ECI Y (equatorial)
  out.set(X, Z, -Y);
  return out;
}

The sun direction is a fixed unit vector in scene space. The Earth fragment shader computes diffuse lighting from the same uSun uniform, so satellite eclipse states computed in JS match the visual day/night terminator on the globe to within the cylindrical shadow approximation.

Brief Upgrades — All Three Implemented

The brief listed three suggested upgrades. All three were completed by the predecessor builder in its first pass.

  • GROUND-TRACK PROJECTION Built as a THREE.Line child of the Earth mesh, so it rotates with Earth. Shows ±1 orbit in the Earth body frame by converting ECI positions to geographic coordinates accounting for Earth's rotation angle at each timestep.
  • TERMINATOR LINE A great circle perpendicular to the sun direction vector, computed by finding two orthogonal basis vectors in the terminator plane and tracing a parametric circle. Added as a child of Earth so it co-rotates, giving the correct day/night boundary.
  • TELEMETRY TICKER Fixed bar at the viewport bottom, cycling through 12 pre-written plausible telemetry dispatches on a 4-second interval. Includes a live UTC clock.

05

Technical Notes

No texture files. The Earth surface is 100% procedural: a 3D FBM hash function painted in the fragment shader generates continent shapes, ocean depths, polar ice caps, city lights on the night side, and a lat/lon graticule — all seam-free because it operates in 3D object space rather than UV space.

Simulation speed. The sim runs at 120× real time (one wall-clock second = two simulation minutes), giving a visible orbit in about 47 seconds of watching. The SPEED constant can be changed without touching the physics.

Coordinate systems. The propagator outputs ECI (Earth-Centered Inertial) coordinates, then maps them to Three.js space (Y-up = ECI Z = north pole; Three.js −Z = ECI Y). The ground-track converts ECI positions back to geographic lat/lon accounting for Earth's rotation angle at each timestep using the sidereal rate W_E = 7.2921 × 10⁻⁵ rad/s.

Instanced rendering. All 12 satellites share one THREE.InstancedMesh. Selected satellite is scaled up by 2.2× via the instance matrix; color is changed per-instance to gold for the selected vehicle.
