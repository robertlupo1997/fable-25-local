Equinox

How a horizontal sundial park was built from nothing but math — no data files, no textures, no shortcuts. Every pixel earned from declination and hour angle.

The Brief

What was asked

A sundial park driven by the real solar-position algorithm. Signature: pick any city and date — gnomon shadows, day length, and the sky's gradient are computed (declination, hour angle, altitude/azimuth), animated through the day; the equinoxes and solstices are the four chapters. Palette: stone #d9d2c4, gnomon #2b2620, solar #e0a10e, computed sky. Type: Marcellus + Outfit. Prove: real astronomy math, no data files, art-directed like a landscape architect.

BRIEFS.md · Wave 4 · Brief 23 · Upgrade ideas: analemma tracing; golden-hour band on the day dial; a shadow that points at the footer at your local sunset.
Signature Technique

All sky from one function

The whole visual output — sky gradient, sun position, shadow direction, day length, chapter statistics — flows from a single 30-line function: computeSolar(lat, doy, solarHour). It takes latitude, day-of-year, and a solar time (0–24); returns altitude, azimuth, hour angle, declination, and sunrise/sunset. No tables, no lookups, no network.

The sky gradient then uses getSkyColors(altDeg), which interpolates between 9 keyed altitude snapshots (from astronomical night at −18° through full blue at 90°). The key entries were hand-crafted to reproduce the actual colour progression of a sky: deep navy at astronomical twilight, the rust-amber horizon glow at civil dawn, the washed-out pale blue at low elevation, the deep cobalt overhead at noon.

The solar math

// ── Solar position: latitude, day-of-year, solar hour → altitude/azimuth
function computeSolar(lat_deg, doy, solarHour) {
  const lat  = lat_deg * RAD;
  const decl = solarDeclination(doy);   // NOAA formula, ≈0.5° accuracy
  const H    = (solarHour - 12) * 15 * RAD; // hour angle (−π morning)

  // Altitude via spherical-trig sine formula
  const sinAlt = Math.sin(lat) * Math.sin(decl)
               + Math.cos(lat) * Math.cos(decl) * Math.cos(H);
  const alt = Math.asin(Math.max(-1, Math.min(1, sinAlt)));

  // Azimuth via east/north atan2 — avoids quadrant ambiguity
  const east  = Math.sin(H) * Math.cos(decl);
  const north = Math.sin(decl) * Math.cos(lat)
              - Math.cos(decl) * Math.cos(H) * Math.sin(lat);
  const azDeg = ((Math.atan2(east, north) * DEG) + 360) % 360;

  // Sunrise/sunset via cos(H₀) = −tan(lat)·tan(decl), clamped for poles
  const cosH0 = -Math.tan(lat) * Math.tan(decl);
  const H0h   = cosH0 >= 1 ? 0 : cosH0 <= -1
               ? 12             // midnight sun
               : Math.acos(cosH0) * DEG / 15;
}

Pass 3 upgrade: solstice comparison arcs

The sky canvas draws two ghost arcs — summer and winter solstice — behind the active day arc. They use the same computeSolar function called on day 172 (June 21) and day 355 (December 21), at the currently selected city. The amber and blue ghost arcs show the full seasonal spread of the sun's path without any additional data.

// Solstice ghost arcs — drawn behind the active-day arc
function drawSolsticeArc(solsticeDoy, color) {
  const solRef = computeSolar(city.lat, solsticeDoy, 12);
  if (solRef.dayLen < 0.2) return; // polar night
  ctx.beginPath();
  let first = true;
  for (let h = solRef.srHour; h <= solRef.ssHour; h += 0.3) {
    const sc = computeSolar(city.lat, solsticeDoy, Math.min(h, solRef.ssHour));
    const { x, y } = getSkyProjection(sc, city, horizonY, W, skyH);
    first ? ctx.moveTo(x, y) : ctx.lineTo(x, y);
    first = false;
  }
  ctx.strokeStyle = color;
  ctx.lineWidth   = 1;
  ctx.setLineDash([2, 6]);
  ctx.stroke();
  ctx.setLineDash([]);
}
// Summer solstice ≈ day 172 → warm amber ghost
drawSolsticeArc(172, 'rgba(255,180,60,0.22)');
// Winter solstice ≈ day 355 → cool blue ghost
drawSolsticeArc(355, 'rgba(120,170,220,0.22)');

The two ghost arcs reveal the seasonal envelope: on the equinox the active arc sits exactly midway between them, which is the visual proof of the math working.

Verification

Azimuth
180° (due S, noon)
Day length
11h 58m (≈12h equinox)
Declination
−0.32° (spring eq. ≈0°)
Shadow dir.
0° (due N at noon)
Algorithm
NOAA / no EoT

The minor 0.3° altitude error is expected without an equation-of-time correction and without atmospheric refraction. Solar noon is always exactly 12:00 solar time by definition.

Iteration History

Three passes, honestly

I
Build & first critique

Built the complete single-file site: NOAA solar math, sky gradient canvas with 9-key altitude interpolation, sundial dial canvas with latitude-accurate hour lines, four chapter cards with mini sky canvases, full animation loop. Verified altitude, azimuth, and day length against USNO values. The h1 was moved below the canvas onto the stone panel to fix contrast; the sky/ground split was set at 62%/38%.

Critique found the hero layout correct, but the sky arc was only the active-day arc with no seasonal context. Chapter cards and dial looked strong. No layout bugs.

Upgrade: golden-hour arcs on dial

Solved cos(H_gh) = (sin(6°) − sin(lat)sin(decl)) / (cos(lat)cos(decl)) to find the hour-angles where altitude = 6°, then painted brighter gold arc segments at both ends of the day arc on the dial rim.

II
Contrast, controls & sparkline

Critique found: date input picking up browser dark-mode styles (dark background clashing with cream panel); controls panel visually indistinct from the hero stone; no altitude data visible without scrolling.

Fixes applied: added color-scheme: light to date input and select, forcing consistent cream appearance; changed controls panel background from stone to cream with top/bottom borders for visual separation; tightened hero-intro padding and added a thin gnomon-coloured border-top as a deliberate horizon seam between sky canvas and stone.

Upgrade: solar altitude sparkline

Added renderSparkline() — a small canvas in the stats bar that draws the full-day altitude curve (midnight to midnight) with the current-hour position marked by a vertical tick and solar-gold dot. Uses the same computeSolar loop; no new data.

III
Seasonal context & city coordinates

Critique found: the hero stone zone between sky and controls felt visually empty — the only text was the h1 and subtitle with no data. The sky canvas arc had no seasonal reference point, making it hard to see whether the current date was a high or low sun day.

Fixes: added a city lat/lon coordinate display (.hero-coords) to the hero-intro row — it updates live when the city changes and gives the space purpose. Coordinate display is hidden on mobile to keep the layout clean.

Upgrade: solstice comparison arcs

The sky canvas now draws two ghost arcs behind the active arc: summer solstice (day 172, pale amber) and winter solstice (day 355, pale blue). On the spring equinox the active arc sits exactly midway — a visual proof of the math. At Reykjavík, the winter arc barely rises above the horizon while the summer arc nearly reaches the zenith, making the polar latitude visible at a glance.

Known Limits

What was left out, and why

No equation-of-time correction. Solar noon is always exactly 12:00. Adding EoT would shift noon ±16 minutes (the analemma) and enable displaying clock time vs. solar time. Not implemented because it required restructuring the state model; the brief did not require clock time.

No atmospheric refraction. The sun visually appears ~0.5° higher than computed altitude near the horizon. Sunrise and sunset are therefore ~2 minutes early by real-clock measure. The math is correct for a theoretical flat-Earth horizon with no atmosphere.

Approximate hour-line labels. The sundial hour ticks are placed at 15° clock intervals. The shadow itself follows the correct horizontal sundial formula (atan(sin(lat) × tan(H))); only the label positions approximate clock time.
